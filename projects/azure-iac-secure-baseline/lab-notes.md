## Session log

**Environment setup.** Worked in Azure Cloud Shell (az + Terraform 1.15.6 pre-installed, already authenticated). Lab files created under `~/lab01-iac` via a bootstrap script. Key learning: Cloud Shell defaults to PowerShell; bash is required for the lab's syntax — switch with `bash` or the shell dropdown.

**Part A — Bicep**
- `az bicep build --file main.bicep` → compiled clean (no errors). Bicep is valid.
- `az deployment sub what-if` → **5 resources to create**, all `+ Create`, no modifies/deletes. Preview matched the intended stack:
  - Resource group `azlab01-lab-rg`
  - NSG `azlab01-nsg` (Deny-All-Inbound, priority 4096)
  - VNet `azlab01-vnet` (10.20.0.0/16, subnet `workload` 10.20.1.0/24, NSG-associated)
  - Storage `azlab01r7bwbsa` (TLS1_2, HTTPS-only, no public blob)
  - Key Vault `azlab01-r7bwb-kv` (RBAC auth, soft-delete 7d)
- Note: the `r7bwb` name suffix is generated by Bicep's `uniqueString()` to guarantee globally-unique storage/KV names.
- `az deployment sub create --name lab01-bicep` → **Succeeded** in 1m34s (mode: Incremental). All 5 resources provisioned.
  - Outputs: RG `azlab01-lab-rg`, storage `azlab01r7bwbsa`, KV `https://azlab01-r7bwb-kv.vault.azure.net/`
- Idempotency check: re-ran what-if → **`5 no change`** (all resources `=`). Confirms Bicep reconciles to desired state; repeat runs are safe/no-op.
- Verified live storage posture: `TLS1_2`, HTTPS-only `true`, public blob `false`. Hardening confirmed on the actual resource, not just in template.
- **Troubleshooting note:** first query returned `https: null`. Root cause was NOT a misconfig — the property is `supportsHttpsTrafficOnly` in Bicep/ARM but `enableHttpsTrafficOnly` in `az storage account show` output. JMESPath returns null for a key that doesn't exist. Lesson: template property names ≠ CLI output property names; a `null` can mean "wrong field name," so verify before assuming the resource is broken.
**Part B — Terraform** (prefix `azlab01b` to avoid collision with the Bicep stack)
- Hit several config issues, each fixed (see troubleshooting log below): single-line `variable` block syntax, multi-line paste dropping newlines in Cloud Shell, `terraform init` needing a rerun after the syntax fix, running commands from the wrong directory, and a deprecated provider argument.
- `terraform init` → installed azurerm v4.80.0.
- `terraform validate` → valid (after fixing deprecated `enable_rbac_authorization` → `rbac_authorization_enabled`).
- `terraform plan -var="name_prefix=azlab01b"` → **`Plan: 7 to add`**.
- **Bicep vs Terraform insight:** same stack = 5 resources in Bicep but 7 in Terraform. Bicep nests the subnet and NSG-association inside the VNet; Terraform models `azurerm_subnet` and `azurerm_subnet_network_security_group_association` as separate first-class resources. Bicep favors nested composition; Terraform favors explicit granular resources.

### Troubleshooting log (Part B)
1. **Single-line block error** — `variable "x" { type = string, default = "y" }` is invalid HCL; block arguments must be newline-separated (commas are only for object/map literals). Fixed by writing multi-line.
2. **Cloud Shell paste dropping newlines** — heredocs pasted into Cloud Shell merged lines (`EOFefault…`). Fix: use a single-line `printf '…\n…'` instead of multi-line heredocs.
3. **"Missing required provider" / "Inconsistent dependency lock file"** — `terraform init` had failed on the earlier syntax error, so no provider was installed. Rerunning `init` after the fix resolved it.
4. **Wrong directory** — ran `terraform plan` from `~` instead of `~/lab01-iac/terraform`; Terraform only sees the initialized config in its own dir.
5. **Deprecated argument warning** — `enable_rbac_authorization` renamed to `rbac_authorization_enabled` in azurerm v4.x (removed in v5.0). Updated to stay forward-compatible.

- `terraform apply -var="name_prefix=azlab01b"` (typed `yes` at the approval gate) → **`Apply complete! Resources: 7 added`**.
  - Outputs: RG `azlab01b-lab-rg`, storage `azlab01ba3d45sa`, KV `https://azlab01b-a3d45-kv.vault.azure.net/`
  - Observations: Terraform built independent resources in parallel (dependency graph), serializing only subnet→VNet. Key Vault took ~2m44s (soft-delete provisioning).
- **Approval-gate difference:** `terraform apply` pauses for a typed `yes`; `az deployment create` applies immediately. Terraform separates plan/approve/apply more explicitly.
**Part C — Break & Diagnose (RCA)** *(this one surfaced naturally, not injected)*
- **Symptom:** the Terraform idempotency re-plan was NOT clean — `Plan: 0 to add, 1 to change, 0 to destroy`, wanting to re-assert the storage `network_rules` block in-place.
- **Investigation:** checked ground truth with `az storage account show ... --query networkRuleSet` → live resource already matched config exactly (`defaultAction: Allow`, `bypass: AzureServices`, no ip/vnet rules). So nothing had actually drifted.
- **Root cause:** known azurerm provider idempotency quirk — an inline `network_rules` block containing only `default_action = "Allow"` + default `bypass` doesn't round-trip cleanly into state, producing a benign phantom "update in-place" on every plan. Not real drift; applying it changes nothing.
- **Fix applied:** removed the redundant inline `network_rules` block (Allow is already the Azure default) → plan returns clean. Production alternative: manage rules via the dedicated `azurerm_storage_account_network_rules` resource, which you'd use anyway when tightening to `default_action = "Deny"` (Lab 07).
- **Lessons:** (1) verify against the live resource before believing a plan diff; (2) explicitly declaring values equal to Azure defaults can invite phantom diffs — keep config minimal; (3) recognizing benign/phantom drift prevents chasing non-issues in production.
- After fix: `terraform plan` → **"No changes. Your infrastructure matches the configuration."** Idempotency confirmed for the Terraform stack too.
**Part D — Teardown**
- `terraform destroy -var="name_prefix=azlab01b"` → **`Destroy complete! Resources: 7 destroyed.`**
  - Note: Key Vault took ~10m to destroy because `purge_soft_delete_on_destroy = true` fully purges (not just soft-deletes) the vault. Tradeoff: slow teardown, but no lingering soft-deleted vault reserving the name/incurring risk.
- `az group delete --name azlab01-lab-rg --yes` → completed (removed all 5 Bicep resources; vault soft-deleted).
- `az keyvault purge --name azlab01-r7bwb-kv` → purged.
- Verify: `az keyvault list-deleted` → **empty**; `az group list` (azlab01*) → **empty**. Environment fully clean, no idle spend.

## Lab 01 — Summary & outcome ✅

**Completed 2026-07-04 in Azure Cloud Shell.** Deployed an identical secure baseline stack (RG + VNet/subnet/NSG + hardened Storage + RBAC Key Vault) twice — once with Bicep (5 resources, subscription-scope), once with Terraform (7 resources, prefix `azlab01b`) — then verified, diagnosed a real issue, and tore everything down.

**Skills exercised:** Bicep modules/params + subscription-scope deployment; `what-if` preview; idempotency; Terraform init/validate/plan/apply/destroy; provider/state model; live-resource verification with `az` + JMESPath; and a genuine RCA on phantom drift.

**Key takeaways**
- Preview before apply, every time (`what-if` / `plan`).
- Idempotency = safe repeat runs; both tools reconcile to desired state.
- Bicep is stateless (reconciles vs. live Azure); Terraform tracks a state file — same stack modeled as 5 vs. 7 resources.
- A `null` in a query can mean "wrong property name," not "broken" (`supportsHttpsTrafficOnly` vs `enableHttpsTrafficOnly`).
- Recognize **phantom/benign drift** (the `network_rules` Allow-default quirk) and don't chase it; verify against the live resource first.
- Always tear down; purging Key Vaults frees the name and stops lingering state.

**Carried into later labs:** remote Terraform state + locking (Lab 02); lock storage to `default_action = "Deny"` + Private Endpoint via the dedicated network-rules resource (Lab 07); disable shared-key access with managed identity (Lab 06).
